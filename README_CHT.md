# academic-ai-workflow

[English](README_EN.md) ｜ **繁體中文**

本專案主要文件為繁體中文。完整的設計原理與架構說明見[工作流架構白皮書](docs/工作流架構白皮書.md)，安裝與操作見[快速上手指南](docs/快速上手指南.md)。只有 claude.ai 網頁版（無 Cowork／Claude Code）的使用者見[在 claude.ai 上使用](docs/在-claude-ai-上使用.md)。

> 我在做的，是將自己的判斷力與領域知識，整包教給 AI，訓練 AI 成為一個不會累的我。
>
> AI 輔助研究的品質與誠信困境，不是技巧問題，而是治理問題。解決方案不在單次對話的層次(prompt)，而在把研究者的判斷力建構為一整套 AI 可讀、可執行、可稽核的架構(harnessing)。
>
> 什麼該進執行層自動化、什麼必須留在使用者手上，這個判斷只有每天親手做研究的你做得出來。

## 問題

生成式 AI 早已進入研究工作。隨手使用（一次提示、一次輸出）時，它會捏造不存在的引用、為真實文獻配上偽造的頁碼與宣稱、把因果語氣講得超出研究設計所能支撐、並在長對話中逐漸退化。常見的兩種回應是全面禁用，或依賴使用者的提示詞技巧。本專案走第三條路，取法公共行政：把 AI 當成一個能力很強但不完全可靠的代理人，然後用組織治理代理人的方式治理它，也就是分工、授權、審計與版本控制。品質來自可靠的流程，而不是可靠的個人，這條原理對 AI 同樣成立。

## 架構：治理基座上的三層

這個分層回答的不是「檔案放在哪裡」，而是「人與 AI 各自把什麼外化給對方」。**規則層**寫給 AI 看（你是誰、該怎麼跟你協作），**思考層**寫給你自己看（這篇論文到底要論證什麼，AI 只是順便讀），**執行層**是人與 AI 共同工作的地方（七道工序）。三層之下是一個**治理基座**：資料夾讀寫規則，決定產出落在哪裡、誰能寫。

1. **規則層：判斷力外化**（`starter-kit/00_專案控制/`）。四份模板把研究者的判斷寫成 AI 可讀的制度：`about-me.md`（目標、時程、優先序規則，並含一份**寫作慣性自我申報**：你條列自己已知的壞習慣，例如「文獻回顧容易無限擴張」，每支 skill 會主動攔截）、`research-identity.md`（研究定位與依掌握深度分五級的理論版圖，讓 AI 校準深度而非從基礎講起）、`writing-standards.md`（論證規則、AI 寫作痕跡的禁止清單、引用誠實性規則，以及一條未達標即須標記而非交付的品質底線）、`portfolio-status.md`（承載時間敏感資訊的動態儀表板）。四份共通一條原則：用你自己的話寫，它們是寫給你常用 AI 的一封信，不是填表格。

2. **思考層：思考的外顯化**（`00_專案控制/paper-structure.md`）。整條管線的中樞，且刻意不是給 AI 的規格檔：你的研究問題（是問題，不是主題）、核心命題、論證路徑、選用的支撐，以及最重要的一欄，你還沒想清楚的卡關點，誠實寫出來。它由工序一協作產出，所有下游工序都錨定在你寫下的思考上；你還沒想清楚的地方，制度要求誠實標記，而不是讓 AI 代想。

3. **執行層：七道工序 skill**（`skills/1-*` 至 `7-*` 為原始碼夾，`skills/packaged/` 為可安裝的 `.skill` 打包檔）。每一支都有明確的輸入合約、輸出格式與品質閘門：

   | # | Skill | 工序 |
   |---|-------|------|
   | 1 | research-design-diagnosis | 論文結構建構（章→節→小節），含承諾閘門：你先給出答案，AI 才提供它的版本 |
   | 2 | literature-search | 以 API 為根據建候選文獻池（OpenAlex／Semantic Scholar），每個 DOI 由腳本驗證，不從模型記憶生候選 |
   | 3 | literature-analysis | 逐篇深度分析；知識檔（與論文無關、可複用）與應用檔（綁定當前論文）分離 |
   | 4 | literature-synthesis | 跨篇理論圖譜，**每個論點都有主人**，標明來源、無無主陳述 |
   | 5 | chapter-drafting | 逐段撰寫，採嚴格引用白名單與 claim 級稽核（連未掛引用的論斷也查） |
   | 6 | review-diagnosis | 審稿與自審品保，評分計畫前置、讓步閾值嚴格（AI 只在事實錯誤、稿內證據、更強反論證時退讓，不因作者堅持而退） |
   | 7 | thematic-analysis | 一手資料（訪談逐字稿、田野筆記）的平行支線：質性分析，附引文溯源稽核（檔名加行號、交付前重讀原文比對） |

   工序 1 至 5 構成線性管線（設計→搜尋→分析→綜整→撰寫）；工序 6 是橫切的品保工序，審其餘各支的產出；工序 7 是平行支線，把主題結果直接餵給撰寫。

**治理基座：資料夾讀寫規則**（`starter-kit/`）。六個資料夾對應研究階段：`00_專案控制/`（控制檔）、`01_文獻/`（文獻）、`02_資料與證據/`（資料與證據）、`03_寫作/`（寫作）、`04_審稿與回應/`（審稿與回應）、`05_輸出/`（輸出）。`00` 至 `04` 在任務執行時唯讀；`05_輸出/` 是 AI **唯一**能寫入的區。所有產出先落在這裡，再由人手動晉升到正式區。權責分離由工作流制度與各工序條文強制，並以「人工搬檔＝簽核」為最後防線。

## 核心重點：一條防捏造鏈

幻覺會往下游繼承：一筆在分析階段混入的捏造宣稱，會被寫進綜整圖譜、再被引進論文正文，每一步都取得更多正當性。因此這裡的防捏造不是某一支 skill 的功能，而是**沿著整條生產鏈的六道防線**：

1. 入口處以 API 為根據搜尋、腳本層驗證 DOI，捏造的文獻進不了文獻池；
2. 一條「素材優先」條款，在分析、綜整、撰寫三層設閘：具體宣稱（數據、頁碼、引句、立場）只能來自實際載入的 PDF，模型記憶只能提示搜尋方向、不得斷言內容；
3. 分析階段的推論稽核，區分「文獻說的」與「AI 推的」；
4. 綜整層的來源歸屬，圖譜中每個論點都帶來源，並顯性回報涵蓋缺口；
5. 撰寫層的 claim 級稽核：連未掛引用的論斷都分類查核，因果宣稱還須通過「推論強度與研究設計是否相符」的檢驗；
6. 質性資料的引文溯源稽核，每則編碼摘錄都錨定檔名與行號，交付前重讀原始逐字稿比對。

共通的哲學：不信任單點、不信任記憶、不讓生成者自己驗證自己。

## 快速上手

```
git clone https://github.com/LoWeiLee/academic-ai-workflow.git
```

1. 把 `starter-kit/` 複製為你的研究工作區。四份控制檔模板都要留在 `00_專案控制/`（skill 會直接讀取，缺檔會使工序中斷），但**首跑只需動筆填其中兩份**：`about-me.md` 與 `research-identity.md`；`writing-standards.md` 沿用預設、`portfolio-status.md` 可留空。
2. 從 `skills/packaged/` 安裝七支 skill（或先只裝 skill 1＋3）。
3. 拿一個真實的小題目跑一遍研究架構建立→單篇文獻分析，體會交接節奏，再逐步擴大。

逐步操作說明見 [docs/快速上手指南.md](docs/快速上手指南.md)。

**環境**：完整的管線形態需要一個持久的工作區資料夾，即 Claude 桌面應用的 Cowork 模式或 Claude Code。**七支 skill 在 claude.ai 網頁版也裝得起來**（Skills 自 Free 方案起可用，於 Settings → Capabilities 開啟 Code execution and file creation 即可），只是從一條管線退化為七把獨立的工具：工序間的交接由你手動搬檔，literature-search 的自動搜尋不可用（沙盒網域白名單不含 OpenAlex／Semantic Scholar），chapter-drafting 的跨對話凍結保護失效。skill 1、3、6 幾乎不受影響。單支用法的完整操作手冊見 [docs/在-claude-ai-上使用.md](docs/在-claude-ai-上使用.md)。

## 誠實的邊界

這套工作流提升的是執行品質；它無法替你解決定義不清的需求或替你做品味判斷，而且它會明白說出來。搜尋過程透明且留有紀錄，但它**不是**系統性文獻回顧工具，不可在論文中如此宣稱。防捏造機制大幅降低風險，但不轉移作者對每一筆引用的最終責任。你的論文永遠是你的論文。
