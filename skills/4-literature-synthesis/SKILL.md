# 跨文件主題綜整技能 v1.1.0

> 本檔守決策層與雙檔規格。論證判斷框架下放 `references/reasoning-framework.md`（四支共用副本，第二階段分析按需載入）；兩份範本（example-literature-synthesis-knowledge／-application）同捆於 `references/`，第零階段強制讀取。雙檔架構（綜整知識檔 paper-agnostic 長期累加正本＋綜整應用檔綁定 paper_id）、三模式（Mode A 全新綜整／Mode B 增量更新／Mode C 既有檔批次拆分，依「主題資料夾內有無綜整知識檔」自動切換，不使用索引檔）為現行設計。Mode C 為一次性遷移工具，置於檔末附錄。變更歷程見 repo CHANGELOG，本檔不嵌補丁／遷移公告。

## 核心目標

將多篇文獻的個別分析，整合為一份**帶有明確文獻歸屬的理論知識圖譜**，並拆為兩份檔案各司其職：

**綜整知識檔（synthesis-knowledge）＝文獻的內在屬性**
這個主題「確信什麼／爭什麼／假設什麼」。paper-agnostic，對使用者的任一論文皆可複用。承載：場域狀態導讀、理論版圖、關鍵概念節點庫、隱性假設清單。它是長期累加的單一正本。

**綜整應用檔（synthesis-application）＝對本研究的定位與寫作指引**
這個主題「對當前這篇論文意味著什麼、缺口在哪、第二章怎麼鋪」。綁定單一 paper_id，可多份。承載：本主題對當前研究的整體意義、研究缺口地圖、文獻回顧架構建議（含概念與共識的章節使用對照）、三句話版本。

**切分核心原則**：paper-bound 內容（對當前研究意涵、概念使用建議、缺口挑選與定位、章節指引）一律歸應用檔；文獻的內在屬性歸知識檔。

這兩份檔案共同滿足兩個使用需求：知識檔是**理解工具**（對主題的系統性全局掌握）兼**寫作素材**（每個論點都標記來源，供 chapter-drafting 直接生成正式引用）；應用檔是**鋪稿藍圖**（為這一篇論文把素材組織成可撰寫的結構）。

## 環境偵測（所有讀取動作之前，最先執行）

本 skill 以 `00_專案控制/` 作為控制檔的預設路徑，此路徑假設一個**持久的工作區資料夾**（Claude 桌面應用的 Cowork 模式，或 Claude Code）。**若該路徑不存在，代表使用者在無工作區環境（例如 claude.ai 網頁版）執行本 skill。此時一律進入「無工作區模式」，不得以「檔案缺失」為由拒絕啟動。**

無工作區模式的三條規則：

1. **控制檔改在別處尋找**。依序查找同名檔案：對話附件 → 專案知識庫（claude.ai 的 Project files）→ 使用者於本次對話明確指定的位置。找到即視同滿足必讀要求，正常推進；三處皆無才回報缺失，並明確告知使用者可透過對話附件或 Project files 提供。**此規則覆蓋本 skill 所有「缺漏即拒絕啟動」的硬閘門條文**——閘門的實質要求是「讀得到該檔的內容」，不是「該檔位於特定路徑」。
2. **產出改以可下載檔案交付**，不嘗試寫入 `05_輸出/` 或任何工作區路徑。晉升流程（審定後移入正式位置）改由使用者手動完成，於交付時一併說明。
3. **跨 session 狀態改由使用者提供**。本 skill 若依賴 `05_輸出/progress-log.md`、既有產出檔或凍結狀態回讀斷點，在無工作區模式下改為請使用者於開場提供前輪產出（附件或貼上內容）；無法取得時，明示「本次為無工作區模式，跨對話狀態不自動續作」後，以本次提供的材料為準推進。

**啟動宣告中必須載明當前模式**：「工作區模式」或「無工作區模式」。

## 寫作原則（不可壓縮的四原則）

**原則一：論點必須有主人**
每一個論點、概念定義、機制描述，都必須標記來自哪篇文獻的哪個位置。禁止無來源論述，即使是綜整性判斷也要說明依據文獻。

**原則二：跨文件整合，不是摘要集合**
「A 說 X，B 說 Y，C 延伸了 A 並挑戰了 B」才是綜整；「A 說 X，B 說 Y，C 說 Z」只是並排摘要。以主題（而非文獻）為組織原則。

**原則三：敘事優先，清單輔助**
每個分析節的主體是連貫段落。清單只用來整理「需要並排比較且有明確引用來源」的結構性資訊。段落說明必須在清單之前。

**原則四：始終對準當前研究**
此原則由**應用檔**承擔。知識檔保持 paper-agnostic（不寫「對使用者的意義」）；所有「對當前研究的意涵」一律寫入應用檔。

**原則五：素材優先（防記憶洩漏）**
綜整的一切具體宣稱（數據、頁碼、引句、作者立場、概念定義）只能來自本 session 載入的知識檔／應用檔（必要時回讀 PDF）；訓練記憶僅供判斷組織方向。素材中查無依據者標記〔記憶未驗證〕，不得寫入正式產出，更不得讓幻覺經綜整層傳入 chapter-drafting。

## 執行程序

### 第零階段：範本讀取儀式（強制優先）

啟動本 Skill 的第一個動作：
1. 以 Glob 檢查本 skill `references/example-literature-synthesis-knowledge.md` 與 `references/example-literature-synthesis-application.md` 是否存在
2. **皆存在**：以 Read 完整讀取兩份，宣告「已讀取知識／應用兩份範本，本次輸出將對齊其格式、深度、語調」
3. **缺其一或皆無**：明確告知使用者，本次將依 SKILL 指引產生，並建議完成後審定保留為對應範本

### 第一階段：模式偵測與輸入準備

**步驟一：路徑與模式偵測（無索引）**
- 知識檔正本路徑：`…/3_文獻/理論基礎/[主題]/[主題]_綜整知識.md`
- 應用檔路徑：`…/3_文獻/應用/[paper-id]/[主題]_綜整應用_[paper-id].md`

啟動某主題綜整時，檢查該主題資料夾（`理論基礎/[主題]/`）內有無 `[主題]_綜整知識.md`：
- **無 → Mode A（全新綜整）**
- **有 → Mode B（增量更新）**，宣告「本主題已有綜整知識檔，進入增量更新模式」

**步驟二：讀取研究脈絡**
- `00_專案控制/about-me.md`、`research-identity.md`、`writing-standards.md`
- 當前專案的 `paper-structure.md`：**必須顯式覆述四項**給使用者確認，寫入應用檔的「脈絡對齊確認」：
  1. 研究問題（至少核心主問題一句）
  2. 理論框架（命名主要框架與關鍵概念）
  3. 本次綜整主題對應的章節編號
  4. 該節在全文中的論述角色（建立共識？處理爭議？推導缺口？）

**步驟三：專案範圍與主題範圍雙重確認**
1. 本次綜整屬於哪個 Paper Project？
2. 針對哪一個主題？**一次只做一個主題**。若要求合併多主題，**拒絕合併**，分開執行、分開輸出。

**步驟四：收集銜接摘要（第一輪讀取）**
讀取本主題所有單篇文獻的**知識檔「核心摘要」＋ 應用檔「應用摘要」**（對齊 literature-analysis 當前版的第一輪介面）。檔名以中文主題關鍵字優先、英文命名次選雙重 Glob。**應用檔 paper_id 匹配**：應用檔僅讀取 frontmatter `paper_id` 與本次綜整目標 paper 一致者；該文獻無本 paper 應用檔時，僅以知識檔核心摘要輸入，並於清單確認時標註「N 篇缺本 paper 應用摘要」——不讀他 paper 的應用檔，防 paper-bound 內容汙染綜整輸入（Mode B 同規則）。
- **檔案發現後強制確認**：列出本次將綜整的文獻清單（檔名＋作者＋年份），向使用者確認範圍是否完整、有無遺漏或需排除。同時 Glob 該主題資料夾的 PDF 與 literature-search 文獻池清單（若存在），回報差集「已分析 N 篇／未分析 M 篇」；未分析者提示先跑 literature-analysis，經使用者確認剔除後才繼續——防未分析文獻靜默隱形。
- **規模護欄**：確認後清單超過 15 篇時，不一次全載。改為分批綜整：每批 ≤10 篇先產「中間綜整摘要」（落檔 `05_輸出/`），全部批次完成後以各批中間摘要＋按需回讀（步驟五）合併為最終雙檔。啟動時向使用者回報篇數與分批計畫。15 篇（含）以內維持一次全載。
- **分批執行規格**（規模護欄觸發時適用）：(1) 切批依文獻清單確認順序均分，主題相近者優先同批，啟動時向使用者展示分批表；(2) 中間檔命名 `[主題]_中間綜整_批N_YYYYMMDD.md`；(3) 中間檔硬性要求——逐論點保留來源標記（作者年份）並預分「知識／應用」兩區，**不得壓縮掉出處**，否則合併期將被迫全量回讀、護欄目的自毀；(4) 合併時以各批中間檔為主要輸入、步驟五按需回讀單篇；(5) 收尾提醒中間檔為過程檔，晉升時勿隨最終雙檔搬移。

**步驟五：深入讀取（第二輪，按需觸發）**
當某論點將作核心共識、某概念將進概念節點庫需原文引句、某論點將置於爭議兩方、某缺口性質有疑義時，回讀該篇**知識檔 Section 1 逐節精華**（literature-analysis 當前版的第二輪介面）。按需執行，但觸發條件出現時不得為省成本略過。

### 第二階段：分析執行

> **階段轉換重申（防長流程遺忘）**：進入分析執行前、以及分批綜整每一批開始前，先重申三條鐵律——① **論點必須有主人**：禁止無來源論述；② **綜整≠摘要並排**：以主題組織、標明 A/B/C 承接與挑戰關係；③ **知識／應用切分**：paper-bound 內容一律歸應用檔，知識檔保持 paper-agnostic。

**強制固定順序**（跨兩檔，節號沿用原圖譜編號、不重編）：
- **知識檔**：frontmatter → 範本對齊確認 → 0. 導讀（場域狀態）→ 1. 理論版圖 → 2. 關鍵概念節點庫 → 4. 隱性假設清單 → 自我檢核（6 項）→ 檔尾簽名
- **應用檔**：frontmatter → 脈絡對齊確認 → 0'. 本主題對當前研究的整體意義 → 3. 研究缺口地圖 → 5. 文獻回顧架構建議 → 6. 三句話版本 → 自我檢核（4 項）→ 檔尾簽名

不得跳節、不得合併、不得調序。

## 切分線規格（哪段進哪一檔）

| 內容 | 歸屬 |
|---|---|
| 0. 導讀「場域狀態段」（共識／爭論焦點／理論厚度） | 知識檔 |
| 0. 導讀「對當前研究的整體意義」段 | 應用檔（置於 0' 開頭，作定位引） |
| 1. 理論版圖（1.1 核心共識／1.2 爭議張力／1.3 演進脈絡） | 知識檔 |
| 1.1／1.2 各論點的「對當前研究意涵」尾 | 應用檔 5.4 |
| 2. 關鍵概念節點庫（標準定義／戰略意圖／理論地位／特定情境適用性） | 知識檔 |
| 2. 概念節點「在當前研究中的使用建議」列 | 應用檔 5.4 |
| 3. 研究缺口地圖（3.1／3.2／3.3）整段 | 應用檔 |
| 4. 隱性假設清單（含特定研究情境的適用性） | 知識檔 |
| 5. 文獻回顧架構建議（5.1／5.2／5.3／5.4） | 應用檔 |
| 6. 三句話版本 | 應用檔 |

**切分理由**：第 3 節（研究缺口地圖）是從「場域知識」推進到「本研究貢獻」的 positioning 論證，3.2 隱性缺口「挑哪些、怎麼框」高度 paper-relative，故整段歸應用檔。知識檔透過 1.2 爭議、1.3 演進末段、4 假設保留可複用的「文獻邊界原料」，供任一 paper 自行讀成各自的缺口地圖。**邊界判準**：凡「文獻沒做到 X」的**事實陳述**（對任一 paper 皆成立的邊界觀察）→ 知識檔（1.3 末段集中列示或第 4 節）；「X 對本研究是機會／該怎麼框」→ 應用檔 3.2。

## 兩檔結構與 frontmatter 規格

### 綜整知識檔（synthesis-knowledge）

```yaml
---
type: synthesis-knowledge
topic: ＜主題A＞
synthesis_version: 1
covers_count: 11
covers_citation_keys:
  - author2020keyword1
  - author2021keyword2
  - "...（本綜整涵蓋的所有單篇 citation_key）"
synthesized_by_skill_version: literature-synthesis-[現行版，見檔尾版本資訊]
created_date: YYYY-MM-DD
last_revised_date: YYYY-MM-DD
revision_log:
  - { date: YYYY-MM-DD, trigger: ＜你的論文＞, change: "首版，N 篇" }
tags:
  - 綜整/＜主題A＞
aliases:
  - ＜主題A＞綜整知識
---
```

- **無 `paper_id`**（paper-agnostic）。
- **`covers_citation_keys` 是載重欄位，不可省，且禁止自行發明**。一律從本主題各單篇知識檔 frontmatter 的 `citation_key` **逐字抄錄**（與 literature-analysis 同源同字面；不得自行照格式規則生成——既有鍵值與格式規則不符時，以逐字抄錄為準）。蒐集不到時回讀該單篇檔，仍無則標記待補，不可臆造。
- `revision_log` 每次 Mode B 追加一行。

### 綜整應用檔（synthesis-application）

```yaml
---
type: synthesis-application
topic: ＜主題A＞
paper_id: ＜你的論文＞
knowledge_note: "[[＜主題A＞_綜整知識]]"
synthesis_knowledge_version: 1
applied_date: YYYY-MM-DD
synthesized_by_skill_version: literature-synthesis-[現行版，見檔尾版本資訊]
tags:
  - paper/＜你的論文＞
  - 綜整/＜主題A＞
---
```

- `knowledge_note` 採 Obsidian wikilink，使「哪些 paper 用過此綜整」由 backlink 自然呈現（取代索引的 applications 欄）。
- `synthesis_knowledge_version` 記錄本應用檔對著哪個知識版本建立——覆寫式單一正本仍可追溯的關鍵欄位。

### 命名規則
- 知識檔：`[主題]_綜整知識.md`（一主題一份）
- 應用檔：`[主題]_綜整應用_[paper-id].md`（一 paper 一份，可多份）

## 兩種日常執行模式（Mode C 見附錄）

### Mode A｜全新綜整
- **觸發**：主題資料夾內查無綜整知識檔
- **讀取**：研究脈絡（含四項覆述）＋ 本主題**全部**單篇文獻的「核心摘要＋應用摘要」（第一輪；超過 15 篇時依步驟四規模護欄分批）＋ 按需回讀知識檔 Section 1（第二輪）
- **產出**：綜整知識檔 ＋ 綜整應用檔兩份 .md
- **docx**：產一份合併版（知識＋應用接續排版），維持 Word 連續閱讀體驗（docx 不含 frontmatter）

### Mode B｜增量更新
- **觸發**：主題資料夾內已有綜整知識檔
- **增量文獻偵測**：讀既有知識檔 `covers_citation_keys`，與該主題資料夾現有單篇知識 md 清單比對；差集＝本次增量文獻
- **讀取**：既有知識檔全文 ＋ 當前 paper 的 paper-structure.md／research-identity.md ＋ **僅增量文獻**的核心摘要＋應用摘要（第一輪），必要時其知識檔 Section 1（第二輪）
- **產出**：
  1. **更新後的完整綜整知識檔全文**（可直接覆寫 vault）＋ 一份**變更摘要**（動了哪幾節、新增哪些 citation_key、有無改寫既有內容）
  2. 該 paper 的**全新綜整應用檔**（`synthesis_knowledge_version` 記錄更新後版本號）
- **加成紀律**：additive 為預設；任何對既有共識／概念／假設的**改寫**，必須在變更摘要明確點名，並在 `revision_log` 追加一行，供使用者審定後才併入
- **frontmatter 更新**：`synthesis_version` +1、`covers_count`／`covers_citation_keys` 補入增量文獻、`last_revised_date`、`revision_log` 追加

## 輸出結構（逐節規格）

> 各節格式硬約束（禁條列、來源標記、層次判斷等）為一貫標準；「歸屬哪一檔」由上方切分線界定。

### 知識檔・0. 導讀：這批文獻在說什麼（僅場域狀態）
2–3 段連貫段落，說明社群關心什麼問題、建立了哪些共識、最主要的爭論焦點，末段說明整體理論厚度。**不含**對特定論文的意義（該段移應用檔 0'）。
- 格式硬約束：禁條列；禁逐篇並排；禁 #### 子標題切分；每段 ≥3 個文獻引用標記。

### 知識檔・1. 理論版圖
先一段連貫文字說明理論發展整體狀態。各子節**移除**「對當前研究意涵」尾（移應用檔 5.4）。
- **1.1 核心共識**：每個共識用【論點】【來源】【論點說明】三欄；【來源】**至少 2 篇**才構成共識；禁表格；禁無引用總括句。
- **1.2 主要爭議／理論張力**：先標示採**層次 A（真實爭議）／B（分析路線張力）／C（無真正分歧）**何者。A 用甲方/乙方；B 用路線一/二並誠實表述非對立；C 一段說明收尾。禁製造不存在的爭議。各爭議的【對當前研究的意涵】移應用檔 5.4。競爭解釋並存時，以 `references/reasoning-framework.md` 的 IBE 四準則評比（非並列）；**知識論狀態標註限本節（1.2）**：對每組爭議／張力標註共識強度（已確立／受支持／初步／推測／爭議），1.1 不另標。
- **1.3 理論演進脈絡**：以段落呈現因果（「因為 X 所以 Y」至少 2 次），階段 3–5 個。末段保留為可複用的場域邊界觀察；paper-bound 定位語移應用檔。

### 知識檔・2. 關鍵概念節點庫
每個概念固定格式：【概念名稱】／【標準定義】（原文引句＋中文說明＋**【戰略意圖】強制欄位**）／【概念的理論地位】／【特定研究情境的適用性說明（例如特定產業、地區、政策場域等，依使用者研究調整）】。**移除**【在當前研究中的使用建議】列（移應用檔 5.4）。
- 二手引用格式：`"原文"（原作者, 原年份, p. X，引自 二手作者, 二手年份, p. Y）`，禁直接寫原作者卻不標「引自」，禁為避麻煩改寫遮蔽原始來源。
- 原文引句 fallback：未保留原文時不得編造，依序觸發第二輪深讀／標記待補。

### 知識檔・4. 隱性假設清單
每個假設：【假設內容】【最依賴此假設的文獻】【若假設不成立的影響】【特定研究情境的適用性判斷】。隱性＝文獻未顯式陳述但論證必須依賴的前提；**2–4 個**。特定研究情境適用性對使用者各論文通用，故留知識檔。

### 應用檔・0'. 本主題對當前研究的整體意義
原導讀第 3 段的「對當前研究意義」內容，置於應用檔開頭作定位引。

### 應用檔・3. 研究缺口地圖
先一段說明整體缺口分布。
- **3.1 文獻自身承認的缺口**：引作者原話／精確描述 + 對當前研究的機會。
- **3.2 綜整後識別的隱性缺口**：跨文件才看得到的盲點，每個**至少 2 篇**邊界共同揭示 + 所需研究設計。
- 3.1／3.2 都必須出現，空則寫明空的理由；兩者皆空須反思主題範圍是否飽和。
- **3.3 當前研究的缺口定位**（最重要）：研究問題落在哪個缺口；缺口性質決定貢獻宣稱（新缺口／實證缺口／情境缺口／**理論橋接**）。必須給明確判斷結論。

### 應用檔・5. 文獻回顧架構建議
- **5.1 建議的論述邏輯**：3–5 步（不少於 3、不多於 6）；文末補「最終章節結構由 chapter-drafting 敲定」。
- **5.2 核心文獻的引用配置**：每篇涵蓋文獻都須歸入**重點引用／背景引用／對比組／不建議引用**四類之一；不建議引用須附具體理由。
- **5.3 待補充的文獻缺口**：主動命名具體補充文獻（作者年份）並對接論述任務；禁無線索推薦；空則明寫「暫無額外補充建議」。
- **5.4 概念與共識的章節使用對照**：收納從知識層搬來的 paper-bound 尾巴，逐項對接具體章節：
  - 概念節點：【概念名稱】→ 可引用於 §X／建議引用表述／需搭配文獻
  - 共識／爭議：【論點或張力】→ 對當前研究的意涵與章節落點

### 應用檔・6. 三句話版本
三段各 1–2 句：已確定知道的事（附 2–3 文獻）／仍不知道的事（附缺口來源）／對當前研究最重要的啟示（第三段給**具體研究策略**）。中文合計 ≤200 字。
