# 研究設計建構技能

## 版本與定位

**版本**：v1.1.0（新增環境偵測與無工作區模式；完整版本歷史見 repo CHANGELOG）
**定位**：全程骨架建構工具，非一次性診斷工具
**設計原則**：階梯式建構、多向工作流、重複調用、與下游 Skills 銜接

> **本檔 references（按需載入）**：`references/adversarial-and-commitment.md`（對抗檢核與 commitment gate）、`references/paper-structure-template.md`（模板正文＋完成清單 canonical 表，第四部分使用）、`references/reasoning-framework.md`（論證與推論框架）。

## 環境偵測（所有讀取動作之前，最先執行）

本 skill 以 `00_專案控制/` 作為控制檔的預設路徑，此路徑假設一個**持久的工作區資料夾**（Claude 桌面應用的 Cowork 模式，或 Claude Code）。**若該路徑不存在，代表使用者在無工作區環境（例如 claude.ai 網頁版）執行本 skill。此時一律進入「無工作區模式」，不得以「檔案缺失」為由拒絕啟動。**

無工作區模式的三條規則：

1. **控制檔改在別處尋找**。依序查找同名檔案：對話附件 → 專案知識庫（claude.ai 的 Project files）→ 使用者於本次對話明確指定的位置。找到即視同滿足必讀要求，正常推進；三處皆無才回報缺失，並明確告知使用者可透過對話附件或 Project files 提供。**此規則覆蓋本 skill 所有「缺漏即拒絕啟動」的硬閘門條文**——閘門的實質要求是「讀得到該檔的內容」，不是「該檔位於特定路徑」。
2. **產出改以可下載檔案交付**，不嘗試寫入 `05_輸出/` 或任何工作區路徑。晉升流程（審定後移入正式位置）改由使用者手動完成，於交付時一併說明。
3. **跨 session 狀態改由使用者提供**。本 skill 若依賴 `05_輸出/progress-log.md`、既有產出檔或凍結狀態回讀斷點，在無工作區模式下改為請使用者於開場提供前輪產出（附件或貼上內容）；無法取得時，明示「本次為無工作區模式，跨對話狀態不自動續作」後，以本次提供的材料為準推進。

**啟動宣告中必須載明當前模式**：「工作區模式」或「無工作區模式」。

## 第一部分｜執行前的準備

### 階段零：必讀脈絡檔案

所有啟動情境都必須先讀取：
- `00_專案控制/about-me.md`：研究發展目標、優先序**規則框架**、AI 角色（穩定身份）、使用者自訂寫作慣性清單（見該檔「寫作慣性自我申報」節，若已填寫）
- `00_專案控制/portfolio-status.md`：**發表組合現況、優先序規則的目前對應、時程事件**（首席參謀戰略評估與資源競爭判斷的動態依據——涉及優先序判斷時必讀）
- `00_專案控制/research-identity.md`：理論版圖、方法論立場
- `00_專案控制/folder-instructions.md`（若存在）：當前研究專案專屬協作規則

### 階段零：模式識別

**模式判定前置（S-0 狀態回讀）**：判定 Mode 前，先 (1) Glob 當前專案 `05_輸出/paper-structure_*` 與 `05_輸出/戰略評估_*`（本 skill 階段輸出）；(2) 讀 `05_輸出/progress-log.md` 末段最近本 skill 條目（若存在），恢復已建構層級、commitment gate 開關狀態、待確認清單；(3) 若 05_輸出 存在比正式位置更新的 paper-structure 修訂版，提示使用者確認要採用哪個版本後再判定——防止版本確認前誤觸 Mode A 全量重建。

**Mode A｜建立新架構**
- 信號：專案資料夾不存在 `paper-structure.md`，或研究者說「新研究」「新計畫」「開始構思」「想寫一篇」「要寫研究計畫書（如國科會）」
- 執行：完整三階段建構

**Mode B｜修訂 / 深化現有架構**
- 信號：已有 `paper-structure.md`，研究者說「繼續建構」「展開第 X 章到節層級」「深入第 X 節到小節」「發現問題要調整」「框架跑掉了」
- 執行：先讀取現有 paper-structure.md，識別當前已建構層級，接續深化或修訂

### 階段零：任務類型識別

**類型 T1｜期刊論文**：5 章（緒論、文獻、方法、結果、結論）；目標 TSSCI / SSCI；小節展開——第 2 章必須、第 3/4 章建議、第 1/5 章可選
**類型 T2｜研究計畫書（如國科會等研究經費申請）**：6 節（研究緣起、研究目的與問題、文獻探討、研究方法、預期結果與貢獻、研究時程）；目標送件爭取經費；小節展開——文獻探討與研究方法必須、其餘可選
**類型 T3｜混合產出（先計畫書後期刊）**：在多篇論文組合中常見；先走 T2 完成計畫書、計畫執行中期產生結果後啟動 T1 反向工作流

> 註：TSSCI 分級與國科會皆為臺灣在地示例，請依你所屬學術體系的期刊分級與補助機構替換。

### 階段零：工作方向識別（關鍵決策）

依任務特性選擇工作方向，**此決策影響整個後續的 Q&A 順序與章節推進順序**。

**方向 D1｜前向工作流**（R → T → M → C → S）
- 適用：全新題目、博論、期刊論文但無既有資料、研究計畫書
- Q&A 順序：核心問題 → 理論 → 方法 → 貢獻 → 戰略；章節推進：第 1 → 2 → 3 → 4 → 5

**方向 D2｜反向工作流**（M + 結果 → T → R → C）
- 適用：量化成果拆分、既有資料二次分析、博士論文章節轉期刊
- Q&A 順序：方法細節 + 已有結果 → 理論對話可能性 → 核心問題（倒推）→ 貢獻定位；章節推進：第 3 → 4 → 2 → 5 → 1

**方向 D3｜計畫書工作流**（缺口 → R/T → M → C）
- 適用：國科會計畫書、合作研究案、新題目的提案階段
- Q&A 順序：研究缺口 → 研究問題 / 理論 → 方法 → 預期貢獻；章節推進：研究緣起 → 目的與問題 → 文獻 → 方法 → 預期結果

### 階段零：範本讀取（若存在）

範本與 SKILL.md 的優先序依第八部分「範本優先原則」執行。

若 `00_專案控制/example-research-design-diagnosis.md` 存在，**強制以 Read 讀取**，作為格式、深度、語調的對齊基準。
若不存在，明確告知研究者：「research-design-diagnosis 尚無範本，本次輸出將依 SKILL.md 現行版指引產生，建議完成後考慮保留為範本。」

### 階段零：來源文件讀取（若有上傳）

若對話中有上傳文件（博論全文、計畫書草稿、前期研究報告、已有量化結果）：

主動讀取並提取以下資訊後再開始 Q&A——**Q&A 只補充無法從文件獲得的決策**：

| 提取項目 | 對應維度 |
|----------|----------|
| 核心研究問題與子問題 | R |
| 援引的理論框架與核心概念 | T |
| 研究方法、資料來源、分析工具 | M |
| 已有結果或發現（反向工作流必備） | 結果 |
| 原文的貢獻宣稱與研究缺口定位 | C |

提取完成後向研究者摘要：「我從這份文件中識別到……（摘要）。以下幾個問題是文件中尚未明確的決策，需要你確認：……」

## 第二部分｜三階段建構架構

> **階段轉換重申（防長流程遺忘）**：每次跨階段推進（章→節、節→小節）或重複調用進入新層級前，先重申三條鐵律——① **commitment gate 不跳過**：三決策點先請使用者一兩句表態再給建議（除非使用者明確關閉）；② **對抗檢核不撤回**：主動挑自家前提，不因使用者堅持而撤回未被有力反駁的挑戰（與 review-diagnosis 讓步閾值不同，此處無讓步條件）；③ **錨定 paper-structure**：輸出前檢查是否偏離既有 paper-structure.md，只動當前層級、局部更新不覆寫整檔。

### 階段一：章層級建構

**目的**：確定大框架的章結構與章間銜接，完成 RTMCS 五維的第一層檢核。

**執行內容**：

1. **依工作方向執行 Q&A**（詳見第三部分各工作方向的 Q&A 腳本）
2. **產出 paper-structure.md 的「各章架構」節骨架**（僅到章的核心任務、字數目標、與前後章的銜接關係）
3. **RTMCS 五維的章層級檢核**：
   - R：核心問題是否明確且可操作？
   - T：援引理論是否真正服務於問題？
   - M：方法是否由問題推導？
   - C：貢獻定位是否清晰？主張強度與設計是否匹配？
   - S：與研究發展目標的吻合度？資源競爭？（依 portfolio-status.md 的當前組合判斷）
4. **標記模糊處**：任何未確認的欄位標記「⚠️ 待確認」，並說明需要釐清什麼
5. **輸出首席參謀戰略評估**：在對話層輸出 + 存檔至 `05_輸出/戰略評估_paper-structure_[日期].md`

**Q&A 彈性**：章層級通常 2-3 輪 Q&A 可完成。若複雜度高，不強制限縮輪次——原則是「核心章框架不清楚時不推進到階段二」。

**階段一結束時的輸出**：
- `05_輸出/paper-structure_[論文名稱]_章層級_[日期].md`
- 對話層：首席參謀戰略評估
- 存檔：`05_輸出/戰略評估_paper-structure_[日期].md`

**輸出後必須明示的正本化步驟（不可省略）**：向使用者說明——「請將確認後的版本**改名為 `paper-structure.md`、搬入 `00_專案控制/`**。skill 2 至 5 一律以 `00_專案控制/paper-structure.md` 這個路徑為錨點讀取它；停留在 `05_輸出/` 的話，下游工序會找不到而中斷。」首次產出時務必主動講，不要假設使用者知道。

### 階段一附加：commitment gate 與對抗檢核

**commitment gate（先表態再給結構）**：三個決策點——選工作方向、定核心 RQ、挑缺口／定貢獻——**出現即觸發，不論所在階段**（選工作方向發生於階段零；RQ 與貢獻的修訂可能發生於 Mode B）。Claude 給建議前，先請使用者一兩句表態自己的答案，再給建議並標示同異。使用者可說「跳過表態」關閉。話術與紀錄格式見 `references/adversarial-and-commitment.md` A 節。設計動機：把判斷權前置，制衡後設漂移與「被 AI 代為框架」風險。

**對抗檢核（devil's advocate，強制）**：階段一 RTMCS 檢核後、戰略評估輸出前，對核心 RQ 與貢獻定位跑一次（前提挑戰／反方視角／frame-lock 三問／因果預設檢），結果填入戰略評估「反方論證與 frame-lock 風險」欄。提問組與紀律見 `references/adversarial-and-commitment.md` B 節；因果與語氣對照見 `references/reasoning-framework.md`。與 review-diagnosis 讓步閾值不同：此處無讓步條件，主動挑自家前提，不因使用者堅持而撤回未被有力反駁的挑戰。

### 階段二：節層級建構

**目的**：每章確定節的切分邏輯、核心任務、字數目標、寫作要點、與 Skills 的銜接切點。

**執行方式**：
- **逐章進行**（不一次展開全部 5 章，避免疲勞）；每次只處理 1-2 章
- 章的推進順序依工作方向決定（見第三部分）

**每章的節層級 Q&A 重點**：
- 本章的主要論述路徑是什麼？（決定節的切分邏輯）
- 每節的核心任務（一句話）？每節的字數目標？
- 每節的寫作要點 / 紀律（對應 writing-standards.md、folder-instructions.md 的哪些規則）？
- 本節可啟動哪個 Skill？（介面合約標記）

**RTMCS 五維的節層級檢核**：R/T/M/C 逐節對齊（S 僅在章層級做）。

**階段二結束時的輸出**：
- paper-structure.md 的「各章架構」節擴展到節層級（累積更新）
- 對話層：標記「當前 paper-structure.md 進度：[哪幾章已完成節層級，哪幾章待推進]」

### 階段三：小節層級建構

**目的**：節之下選擇性展開小節的論述方向、論證邏輯、段落骨架提示。

**執行方式**：
- **按需逐節進行**。原則上必須展開到小節者：第 2 章所有節、第 3 章方法學核心節、第 4 章所有節。按需展開者：第 1 章、第 5 章、第 3 章總論性節。

**每節的小節層級 Q&A 重點**：論述順序、每小節核心論點、論證邏輯、關鍵文獻或證據、段落骨架（可選）。

**RTMCS 檢核**：R（小節論點是否支撐節任務）/ T（理論論證是否嚴密）/ C（是否支撐貢獻定位某面向）。

**文獻白名單協作（第 2 章小節建構完成後，標記 literature-search 之前必做）**：

針對剛完成小節建構的節，主動執行以下步驟：
1. 基於已確立的 RQ、理論框架、論述順序，從訓練記憶推薦該節的奠基文獻與近期關鍵文獻（3–8 篇），每篇附：作者年份、標題、推薦理由（一句話）、預估 DOI
2. 向使用者說明：「以下是我認為這節應優先掌握的關鍵文獻，確認後可填入 paper-structure.md 的經典文獻白名單欄位，literature-search 執行時這些文獻將跳過 rubric 評分直接納入。」
3. 使用者確認（可接受 / 替換 / 增減）後，將確認清單寫入 paper-structure.md 的**「經典文獻白名單（DOI）」**欄位——欄位結構以 `references/paper-structure-template.md` 該表為單一權威，本檔不重述欄位
4. 若 DOI 不確定，標記「⚠️ DOI 待 literature-search 驗證」，literature-search 的 doi_verify.py 會在階段一 Step 2 執行驗證

> **注意**：此步驟是補強「你第一次碰到的理論領域」覆蓋率的關鍵機會。Claude 的訓練記憶在這裡比 API 搜尋更能識別該理論的奠基傳統與主流學者。

**「此節可啟動 Skill X」介面合約標記**：
- 第 2 章某節達到小節層次 → 標記「✅ 可啟動 literature-search 文獻搜尋」
- 第 2 章某節 literature-search 完成（`_metadata.json` 已產出，且 `pending_zotero_transfer.md` 的「PDF 已晉升至 vault」欄已勾選）→ 標記「✅ 可啟動 literature-analysis 逐篇文獻分析」
- 第 2 章某節 literature-analysis 完成 → 標記「✅ 可啟動 literature-synthesis 跨文件綜整」
- 第 2 章某節 literature-synthesis 綜整完成 + 論述邏輯確定 → 標記「✅ 可啟動 chapter-drafting 章節撰寫」
- 其他章節達小節層級 → 標記「✅ 可啟動 chapter-drafting 章節撰寫」

**階段三結束時的輸出**：
- paper-structure.md 擴展到小節層級（累積更新）
- 對話層：標記下一步建議（啟動下游 Skills / 繼續深化其他節 / Mode B 修訂）
- 對話層：**若 `00_專案控制/paper-structure.md` 尚不存在**（即目前只有 `05_輸出/` 的版本），再次提醒使用者完成正本化——改名為 `paper-structure.md` 搬入 `00_專案控制/`，否則上表標記的下游 Skills 一律啟動不了

## 第三部分｜三種工作方向的 Q&A 腳本

### 方向 D1｜前向工作流

**章節推進順序**：第 1 → 2 → 3 → 4 → 5

**階段一 Q&A 核心問題清單**（依情境選 3-5 個）：
- **R**：你想回答的核心問題是什麼？問題類型是描述性／解釋性／規範性？
- **T**：打算援引哪些理論？是核心分析框架還是工具性輔助？多個理論是互補、競爭、還是整合？
- **M**：選擇此方法的理由？由研究問題推導嗎？資料已有還是需蒐集？
- **C**：要與哪個文獻對話、填補什麼缺口（理論／方法／實證）？主張強度是探索性框架、初步理論建構、還是假說驗證？
- **S**：在當前研究組合中的位置（依 portfolio-status.md）？有無時序衝突？目標期刊是否匹配貢獻強度？

### 方向 D2｜反向工作流

**章節推進順序**：第 3 → 4 → 2 → 5 → 1

**階段一 Q&A 核心問題清單**（依情境選 3-6 個）：
- **M（詳細）**：用了什麼方法？樣本規模、時間範圍、抽樣方式？變項如何操作化？統計分析方法與分析單位？
- **結果（反向專屬）**：主要發現？哪些假設被支持／不被支持？哪些結果意外或超越既有文獻？效果量／解釋力多強？
- **T**：這些結果可與什麼理論對話？意外結果可推進哪些理論邊界？是否挑戰主流理論？
- **R（倒推）**：從結果倒推可回答的核心問題？哪種陳述最能突顯結果價值？是否需聚焦？
- **C**：貢獻定位？與哪個文獻直接對話？實務意涵？

**反向工作流特別提醒**：誠實論述方法論軌跡（不得將倒推包裝成「初始即確立的理論框架」，對齊 writing-standards.md）；避免「過度理論化」（難解釋的結果保留為邊界條件，不硬套理論）。

### 方向 D3｜計畫書工作流

**章節推進順序**：研究緣起 → 目的與問題 → 文獻 → 方法 → 預期結果

**階段一 Q&A 核心問題清單**（依情境選 4-6 個）：
- **缺口**：目前文獻的研究缺口？為何重要？是理論／方法／實證缺口？實務層面是否有對應問題？
- **R**：基於缺口，核心研究問題與子問題？可操作性（計畫書審查關鍵）？
- **T**：擬援引理論框架？與缺口的對話關係？理論掌握度（工具性援引需聲明深化計畫）？
- **M**：擬採用方法？為何能回答研究問題？可行性（資料可得性、經費、時程）？
- **C**：預期理論／實務貢獻？發表規劃（幾篇、哪些期刊）？對使用者學術發展的影響？

**計畫書工作流特別提醒**：文獻探討以確立缺口為目的（較短）；預期結果要誠實（過度承諾結案受挫）；時程保留 buffer（質性訪談困難、量化樣本招募困難）。

## 第五部分｜重複調用機制

research-design-diagnosis 不是一次性啟動，而是在以下情境重複被調用：

1. **啟動新章節的節層級建構時**：「幫我展開第 3 章到節層級」
2. **啟動小節層級建構時**：「幫我展開 2.1 節到小節層級」
3. **Mode B 修訂**：「3.4 節的方法論論述結構需要重新檢視」
4. **chapter-drafting 寫作發現結構不夠時**：「第 2.2 節寫作時發現論述邏輯不清楚，需要回 research-design-diagnosis 補強」

重複調用時的執行原則：
- **只針對當前任務層級執行**，不重新跑整個 RTMCS；但修訂涉及核心 RQ 或貢獻定位時，該決策點的 commitment gate 與對抗檢核仍須觸發（僅針對該點）
- **paper-structure.md 的更新採「局部更新」**，不覆寫整個檔案；且所有修訂（含階段二、三的累積更新）一律先產出於 `05_輸出/` 修訂版，經使用者確認後才正式採用，不就地改動唯讀區正本
- **在 progress-log.md 記錄每次調用**（統一位於當前專案 `05_輸出/progress-log.md`，S-0 狀態留痕入口：啟動回讀、變更即追加），包含調用目的、推進內容、下一步、commitment gate 開關狀態（若使用者曾說「跳過表態」）

## 第六部分｜與下游 Skills 的介面合約

research-design-diagnosis 產出的 paper-structure.md 達到特定層次時，觸發下游 Skills 的啟動條件：

### 新增合約條文：文獻分析完成清單的回寫

literature-analysis **每完成一篇單篇分析後**，回寫 paper-structure.md 的「文獻分析完成清單」新增一列。**欄位 canonical 定義以 `references/paper-structure-template.md`「文獻分析完成清單」表為單一權威**——本 skill 與 literature-analysis 均對齊該範本，不各自重述欄位以免 schema 漂移；literature-analysis 實際回寫時以目標 paper-structure.md 的既有 schema 為準（見其「文獻分析完成清單回寫」節的對齊規則與 fallback）。此清單是 literature-synthesis 啟動時確認「哪些文獻已備分析、可調用」的單一真實來源。

### 首次建立範本的時機

首次完整走完三階段建構並產出完整 paper-structure.md 後，建議保留一份代表性案例為 `00_專案控制/example-research-design-diagnosis.md`（依實際走過的工作方向命名，如「反向工作流範本」）。未來走過其他工作方向（如前向）後，可另存對應範本（命名：`example-research-design-diagnosis-前向.md`，與既有範本並存）。
