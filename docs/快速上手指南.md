# 快速上手指南：從 clone 到跑通第一次文獻分析

本指南帶你在半天內完成安裝，並用一個真實的小題目跑通「文獻搜尋 → 逐篇分析」的最小鏈條。架構與設計原理請先讀[工作流架構白皮書](工作流架構白皮書.md)。不必逐字讀完，但至少讀第二、三節，理解三層架構與「05_輸出 是唯一寫入區」的治理邏輯，操作時才知道自己在做什麼。

## 事前準備

你需要：一個可執行 Claude skill 的環境（Claude 桌面應用的 Cowork 模式，或 Claude Code）；Zotero（免費文獻管理軟體，用於取得 PDF）；一個雲端同步資料夾（OneDrive、Dropbox 等，建議但非必要，版本歷史是這套工作流的最後一道保險）。

## 第一步：取得套件並建立工作區

```
git clone https://github.com/LoWeiLee/academic-ai-workflow.git
```

不熟 git 的話，直接在 GitHub 頁面點「Code → Download ZIP」下載解壓即可，效果相同。

接著把 `starter-kit/` 整個資料夾複製到你打算長期使用的位置（例如雲端同步資料夾內），改名為你的工作區名稱。工作區內是完整的資料夾骨架：

| 資料夾 | 用途 | 讀寫規則 |
|---|---|---|
| `00_專案控制/` | 四份控制檔（你是誰、怎麼協作） | 任務執行時唯讀 |
| `01_文獻/` | 文獻庫（PDF 與分析定稿） | 唯讀，晉升由你手動執行 |
| `02_資料與證據/` | 一手資料（訪談稿、數據） | 唯讀 |
| `03_寫作/` | 章節定稿 | 唯讀，定稿由你手動移入 |
| `04_審稿與回應/` | 審稿意見與回應 | 唯讀 |
| `05_輸出/` | AI 所有產出的唯一寫入區 | AI 可寫 |

01–05 各資料夾內有 README 說明細部規則（00_專案控制/ 的規則直接寫在四份控制檔內）。記住一條主線就好：AI 只能寫 `05_輸出/`，其他地方的每一個檔案都是經你的手放進去的。

## 第二步：填寫四份控制檔

這是整個安裝過程中最重要、也最不能敷衍的一步，控制檔的品質直接決定 AI 協作的品質。四份模板都在 `00_專案控制/`，每個欄位附填寫指引與虛構範例句（範例是給你看格式的，內容請全部換成你自己的）。

填寫順序建議：先填 `about-me.md`（你的目標、時程壓力、每週可投入研究的時數、多任務時的優先序規則），其中「寫作慣性自我申報」一節請對自己誠實，你已知的寫作壞習慣（文獻回顧收不住？過度抽象化？開頭寫不出來？）寫得越具體，AI 的攔截就越準。再填 `research-identity.md`（研究定位與理論版圖，按掌握深度分五級，這決定 AI 對你說話的深度）。然後過一遍 `writing-standards.md`（多數條文可直接沿用，個人風格庫可先留空，日後累積）。最後把 `portfolio-status.md` 填上你目前的論文進度。

初次填寫抓大放小即可，用兩三週後回頭修訂一輪，會準得多。

## 第三步：安裝七支 skill

`skills/packaged/` 內有七支打包好的 `.skill` 檔（即以 zip 格式打包的 skill 資料夾），`skills/1-*` 至 `7-*` 是對應的原始檔。安裝方式依你的環境：使用 Claude 桌面應用（Cowork）者，直接匯入 `.skill` 檔；使用 Claude Code 者，將 `.skill` 解壓後的資料夾放入 skills 目錄。細節以你所用版本的官方說明為準。

不必一次裝七支。最小起步只需要兩支：**literature-search（文獻搜尋）與 literature-analysis（逐篇分析）**。等你熟悉節奏後，再依需求加裝其餘五支。

## 第四步：跑通第一次「搜尋 → 分析」

挑一個你真實關心、但範圍很小的題目來試，例如你下一篇論文某一節需要的某個概念。不要用假題目，假題目測不出人機分工的手感。

**（一）給 AI 一個最小的結構錨點。** 文獻搜尋以 `00_專案控制/paper-structure.md`（論文結構檔）為判讀依據。另提醒：執行搜尋腳本前，先把 `scripts/` 內三支腳本的 `MAILTO = "YOUR_EMAIL@example.com"` 改成你的 email（OpenAlex 禮貌池要求，不改也能跑，但請尊重 API 方的規範）。完整的結構檔由 research-design-diagnosis 工序建構，但第一次試跑可以從簡：對 Claude 說「我要為〈某主題〉建文獻池，研究問題是……，理論框架是……」，依它的引導把最小必要資訊補齊。

**（二）啟動文獻搜尋。** 對 Claude 說「幫我找〈主題〉的文獻，建文獻池」。流程會經過幾個由你裁決的關卡：確認種子文獻、確認檢索關鍵字、裁決分桶結果（納入／暫存／排除）。你會注意到 AI 推薦的每一筆文獻都經過 DOI 驗證、每一步都記入搜尋日誌。這不是繁文縟節，是這套工作流的核心保證：**候選池裡不會有模型憑記憶捏造的文獻**。

**（三）用 Zotero 取得 PDF。** 搜尋完成後會產出一份 DOI 清單（`zotero_import_dois.txt`）。打開 Zotero，用「透過識別碼新增」（Add by Identifier）貼入 DOI 批次匯入，Zotero 會盡力抓取 PDF；抓不到的，依清單提示手動補齊。PDF 取得由你在本機完成，工作流不代抓，這同時是技術限制與著作權邊界。

**（四）啟動逐篇分析。** 把 PDF 放入文獻庫後，對 Claude 說「幫我分析這篇文獻」。分析完成後你會拿到兩個檔案：**知識檔**（這篇文獻本身說了什麼，與你的論文無關，永久可複用）與**應用檔**（它對你這篇論文有什麼用）。這個雙檔分離是本工作流的招牌設計：日後換一篇論文用到同一篇文獻，知識檔直接複用，只需補一份新的應用檔。

**（五）審定與晉升。** 產出落在 `05_輸出/`。讀過、確認品質無誤後，由你親手把定稿移入 `01_文獻/`。這個「搬檔案」的動作就是你的簽核，AI 生產，你蓋章。

跑完這五步，你已經體驗了這套工作流的全部核心邏輯：結構錨點、人機檢查點、引用真實性防線、雙檔架構、產出晉升。之後的擴展（跨篇綜整、章節撰寫、審稿診斷）都是同一套節奏的延伸。

## 常見問題

**搜尋回報「環境阻擋」或零筆結果？** 檢查你的執行環境是否允許連外網路（OpenAlex 與 Semantic Scholar API）。工作流會明確區分「環境壞了」與「真的沒有文獻」，不會拿記憶充數；若環境受限，依提示改走手動匯入模式。

**可以跳過控制檔直接用 skill 嗎？** 技術上部分 skill 會降級運作，但你會失去這套工作流最有價值的部分，AI 對「你」的校準。強烈不建議。

**產出可以直接用在論文裡嗎？** 分析與綜整產出是你的研究素材，章節撰寫產出是逐段經你確認的草稿，但每一筆引用的最終責任在作者。工作流大幅降低捏造風險，不轉移這份責任。

**七支一定要照順序用嗎？** 線性順序是「設計 → 搜尋 → 分析 → 綜整 → 撰寫」，審稿診斷橫切各階段皆可用，質性主題分析是處理一手資料的平行支線。從中段切入也可以（例如已有文獻池、直接從分析開始），各 skill 會檢查前置輸入、缺什麼就告訴你。
