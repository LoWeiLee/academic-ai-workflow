# 快速上手指南：從 clone 到跑通第一次文獻分析

本指南帶你在半天內完成安裝，並用一個真實的小題目跑通「研究設計 → 逐篇分析」的最小鏈條。架構與設計原理請先讀[工作流架構白皮書](工作流架構白皮書.md)。不必逐字讀完，但至少讀第二、三節，理解三層架構與「05_輸出 是唯一寫入區」的治理邏輯，操作時才知道自己在做什麼。

## 事前準備

**環境**。這套工作流有兩種用法，先選一種：

**A. 管線用法（本指南的主線）**：七道工序共用一個**持久的工作區資料夾**，前一支的產出自動落在下一支讀得到的位置。需要 Claude 桌面應用的 **Cowork** 模式，或 **Claude Code**模式，以下所有步驟以此為預設。

**B. 單支工具用法（claude.ai 網頁版）**：七支 skill 在 claude.ai 都裝得起來，**Skills 從免費方案起就可用**（Settings → Capabilities 開啟 Code execution 即可）。你得到的不是一條管線，而是七把獨立的工具：今天開一個對話討論論文架構，明天開另一個對話分析一篇文獻，工序之間的交接由你手動搬檔。**這條路徑不需要付費、不需要桌面應用**，而且 research-design-diagnosis（研究設計）、literature-analysis（逐篇分析）、review-diagnosis（審稿診斷）這三支的體驗與完整環境幾乎沒有差別。

簡單說，這本來就是基於學術研究工作流所設計的，A 模式只是用 AI 工具，把各單點工作串聯起來，B 模式則是在原本的每個單點作業下，加入 AI 輔助。

| | claude.ai（單支工具） | Cowork／Claude Code（管線） |
|---|---|---|
| 安裝七支 skill（含 references／scripts） | 可 | 可 |
| 讀控制檔、PDF、逐字稿 | 可（Project files／對話附件） | 可（工作區資料夾） |
| 產出 .md／.docx／.xlsx | 可（下載） | 可（寫入 `05_輸出/`） |
| 工序間交接 | **手動搬檔** | 自動（`05_輸出/`） |
| 跨對話續作（`progress-log.md`、章節凍結段） | 不可 | 可 |
| literature-search 自動搜尋 | 不可（沙盒白名單不含 OpenAlex／S2） | 可 |

**走 B 路徑的人請改讀 [在 claude.ai 上使用](在-claude-ai-上使用.md)**，那份文件是單支用法的完整操作手冊（含開場宣告模板與七支的單支使用卡）。本指南以下的第 3.5 步、第四步（二）（三）（五）、第五步與常見問題第一則都預設你有工作區資料夾。

**其他**：Zotero（免費文獻管理軟體，用於取得 PDF）；一個雲端同步資料夾（OneDrive、Dropbox 等，建議但非必要，版本歷史是這套工作流的最後一道保險）。

## 第一步：取得套件並建立工作區

```
git clone https://github.com/LoWeiLee/academic-ai-workflow.git
```

不熟 git 的話，直接在 GitHub 頁面點「Code → Download ZIP」下載解壓即可，效果相同。

接著把 `starter-kit/` 整個資料夾複製到你打算長期使用的位置（例如雲端同步資料夾內），改名為你的工作區名稱。工作區內是完整的資料夾骨架：

| 資料夾 | 用途 | 讀寫規則 |
|---|---|---|
| `00_專案控制/` | 四份控制檔（你是誰、怎麼協作） | 任務執行時唯讀（唯一例外：literature-analysis 依其協議回寫 `paper-structure.md` 的「文獻分析完成清單」） |
| `01_文獻/` | 文獻庫（PDF 與分析定稿） | 唯讀，晉升由你手動執行 |
| `02_資料與證據/` | 一手資料（訪談稿、數據） | 唯讀 |
| `03_寫作/` | 章節定稿 | 唯讀，定稿由你手動移入 |
| `04_審稿與回應/` | 審稿意見與回應 | 唯讀 |
| `05_輸出/` | AI 所有產出的唯一寫入區 | AI 可寫 |

01–05 各資料夾內有 README 說明細部規則（00_專案控制/ 的規則直接寫在四份控制檔內）。工作區根目錄另有一份 `CLAUDE.md`，把上表的治理規則寫成 AI 開場即讀的版本：掛載工作區後它會自動載入，因此即使你沒有觸發任何 skill、只是直接對話請 AI 動檔案，唯讀分區與晉升流程一樣有效。記住一條主線就好：AI 只能寫 `05_輸出/`，其他地方的每一個檔案都是經你的手放進去的。

## 第二步：控制檔 四份就位，兩份必填

這是整個安裝過程中最重要、也最不能敷衍的一步，控制檔的品質直接決定 AI 協作的品質。四份模板都在 `00_專案控制/`，每個欄位附填寫指引與虛構範例句（範例是給你看格式的，內容請全部換成你自己的）。

先分清楚兩件事：

**四份都必須留在 `00_專案控制/`。** skill 會直接讀取它們，缺檔會導致工序中斷——例如 chapter-drafting 讀不到 `writing-standards.md` 就會停下來要你補齊。所以不要刪、不要搬走。

**但首跑只需要動筆客製化其中兩份**：

1. **`about-me.md`（必填）**：你的目標、時程壓力、每週可投入研究的時數、多任務時的優先序規則。其中「寫作慣性自我申報」一節請對自己誠實——你已知的寫作壞習慣（文獻回顧收不住？過度抽象化？開頭寫不出來？）寫得越具體，AI 的攔截就越準。
2. **`research-identity.md`（必填）**：研究定位與理論版圖，按掌握深度分五級。這決定 AI 對你說話的深度，是它不會把你當大學生講基礎的原因。
3. `writing-standards.md`（沿用預設即可）：模板已帶完整的禁止清單、引用格式與學術中文語氣條文，過一遍、不必改。「個人風格庫」先留空，日後累積。
4. `portfolio-status.md`（可留空）：發表組合的動態儀表板。只有一篇論文在跑時沒什麼組合可談，等你真的多線並行再補。

初次填寫抓大放小即可，用兩三週後回頭修訂一輪，會準得多。

## 第三步：安裝 skill

`skills/packaged/` 內有七支打包好的 `.skill` 檔（即以 zip 格式打包的 skill 資料夾），`skills/1-*` 至 `7-*` 是對應的原始檔。

**使用 Claude 桌面應用（Cowork）者**：開啟 Claude 桌面應用 → Settings（設定）→ Capabilities（功能）→ Skills → Upload skill → 選取 `.skill` 檔。**使用 Claude Code 者**：將 `.skill` 解壓後的資料夾放入 skills 目錄。**使用 claude.ai 網頁版者**：Settings → Capabilities 開啟 Code execution → Customize → Skills → 上傳 `.skill` 檔（`references/` 與 `scripts/` 隨包上傳，不需拆解）。介面選項可能隨版本調整，細節以你所用版本的官方說明為準。

安裝後不需要輸入任何指令，直接在對話中以自然語言觸發即可（例如「幫我分析這篇文獻」）。

不必一次裝七支。**最小起步只需要兩支：research-design-diagnosis（研究設計）與 literature-analysis（逐篇分析）**。這兩支構成一條完整的閉環：先把研究結構想清楚，再讓 AI 深度分析你手上的文獻。等你熟悉節奏後，再依需求加裝其餘五支（自動建文獻池的 literature-search 見第五步）。

## 第 3.5 步：把工作區交給 Claude

**這一步不能跳過**：Claude 預設看不到你電腦裡的任何檔案。你必須明確把第一步建立的工作區資料夾交給它，它才讀得到 `00_專案控制/`、也才寫得進 `05_輸出/`。

在 Claude 桌面應用進入 Cowork 模式後，於對話介面的資料夾入口選取你的工作區資料夾（即第一步複製並改名的那個資料夾，**要選整個工作區，不是其中某個子資料夾**）。介面按鈕名稱可能隨版本調整，找的是「選擇／新增資料夾」這類入口；以你所用版本的官方說明為準。使用 Claude Code 者，在工作區資料夾內啟動即可。

**每次開新對話都要確認工作區已連接。** 這是新手最常見的卡點：控制檔填得再好，工作區沒接上，AI 就是讀不到，整條鏈條無法啟動。

## 第四步：跑通第一次「研究設計 → 逐篇分析」

挑一個你真實關心、但範圍很小的題目來試，例如你下一篇論文某一節需要的某個概念。不要用假題目，假題目測不出人機分工的手感。

**（一）啟動研究設計，把結構想清楚。** 對 Claude 說「我要開始寫一篇關於〈某主題〉的論文，幫我建結構」。research-design-diagnosis 會以階梯式（章→節→小節）帶你走，過程中有一道 commitment gate：在工作方向、核心研究問題、貢獻宣稱這三個決策點上，**你必須先講出自己的答案，AI 才給它的版本**。這個順序不可倒置，先看 AI 的答案再「選擇」，你的框架已經被錨定了。

**（二）把結構檔正本化。** 工序一的產出會落在 `05_輸出/`，檔名類似 `paper-structure_〈論文名稱〉_章層級_〈日期〉.md`。**讀過、確認無誤後，請把它改名為 `paper-structure.md`，搬入 `00_專案控制/`。** 這一步是必要的：`paper-structure.md` 是整條管線的中樞，工序 2 至 5 都以 `00_專案控制/paper-structure.md` 這個路徑為錨點去找它，沒搬過去，下游工序會找不到而中斷。（`00_專案控制/paper-structure.md` 初始不存在是正常的，它本來就由工序一產出。）

**（三）用 Zotero 取得 PDF。** 找幾篇你這一節真的要讀的文獻，用 Zotero 匯入取得 PDF，放進 `01_文獻/`。PDF 取得由你在本機完成，工作流不代抓，這同時是技術限制與著作權邊界。（若你想讓 AI 幫你建候選文獻池，見第五步。）

**（四）啟動逐篇分析。** 對 Claude 說「幫我分析這篇文獻」。分析完成後你會拿到兩個檔案：**知識檔**（這篇文獻本身說了什麼，與你的論文無關，永久可複用）與**應用檔**（它對你這篇論文有什麼用）。這個雙檔分離是本工作流的招牌設計：日後換一篇論文用到同一篇文獻，知識檔直接複用，只需補一份新的應用檔。

**（五）審定與晉升。** 產出落在 `05_輸出/`。讀過、確認品質無誤後，由你親手把定稿移入 `01_文獻/`。這個「搬檔案」的動作就是你的簽核，AI 生產，你蓋章。

跑完這五步，你已經體驗了這套工作流的全部核心邏輯：結構錨點、人機檢查點、引用真實性防線、雙檔架構、產出晉升。之後的擴展（文獻搜尋、跨篇綜整、章節撰寫、審稿診斷）都是同一套節奏的延伸。

## 第五步（選配）：加裝 literature-search 自動建文獻池

熟悉節奏後，literature-search 是投資報酬率最高的下一支：它幫你省去檢索文獻時的第一步篩選。注意它需要可執行腳本與連外網路的環境，且 token 消耗較高。

安裝後，對 Claude 說「幫我找〈主題〉的文獻，建文獻池」。它會以 `00_專案控制/paper-structure.md` 為判讀依據（所以第四步的正本化不能省），流程會經過幾個由你裁決的關卡：確認種子文獻、確認檢索關鍵字、裁決分桶結果（納入／暫存／排除）。你會注意到 AI 推薦的每一筆文獻都經過 DOI 驗證、每一步都記入搜尋日誌。這不是繁文縟節，是這套工作流的核心保證：**候選池裡不會有模型憑記憶捏造的文獻**。

搜尋完成後會產出一份 DOI 清單（`zotero_import_dois.txt`）。打開 Zotero，用「透過識別碼新增」（Add by Identifier）貼入 DOI 批次匯入，Zotero 會盡力抓取 PDF；抓不到的，依清單提示手動補齊。接著就回到第四步（四）的逐篇分析。

**進階選配（可略）**：OpenAlex 有一個「禮貌池」（polite pool）機制，帶上聯絡 email 的請求會得到較穩定的服務品質。腳本內的 `MAILTO` 預設為佔位符，**不改也能跑**；若要設定，需修改你實際安裝的那份 skill 內 `scripts/` 三支腳本的 `MAILTO` 值（改 repo 內的原始檔不會影響已安裝的打包版）。腳本偵測到佔位符時會印一行提示，但不會中斷執行。

## 常見問題

**AI 說找不到 `00_專案控制/` 或看不到我的檔案？** 十之八九是工作區沒連接（見第 3.5 步）。每次開新對話都要重新確認。

**skill 說找不到 `paper-structure.md`？** 這個檔案初始不存在是正常的，它由工序一（research-design-diagnosis）產出。產出後落在 `05_輸出/`，請確認你已把它改名搬入 `00_專案控制/`（見第四步（二））。若你只想先試 literature-search、不跑工序一，也可以手寫一份簡化版：至少寫出研究問題與理論框架兩欄。

**搜尋回報「環境阻擋」或零筆結果？** 檢查你的執行環境是否允許連外網路（OpenAlex 與 Semantic Scholar API）。工作流會明確區分「環境壞了」與「真的沒有文獻」，不會拿記憶充數；若環境受限，依提示改走手動匯入模式。

**可以跳過控制檔直接用 skill 嗎？** 技術上部分 skill 會降級運作，但你會失去這套工作流最有價值的部分，AI 對「你」的校準。強烈不建議。

**產出可以直接用在論文裡嗎？** 分析與綜整產出是你的研究素材，章節撰寫產出是逐段經你確認的草稿，但每一筆引用的最終責任在作者。工作流大幅降低捏造風險，不轉移這份責任。

**我只有 claude.ai，沒有 Cowork／Claude Code，能用嗎？** 能。七支都裝得起來（Free 方案即可），只是變成「七把獨立的工具」而非一條管線：工序間的交接由你手動搬檔，literature-search 的自動搜尋不可用，chapter-drafting 的跨對話凍結保護失效。research-design-diagnosis、literature-analysis、review-diagnosis 這三支幾乎不受影響。完整操作見 [在 claude.ai 上使用](在-claude-ai-上使用.md)。

**七支一定要照順序用嗎？** 線性順序是「設計 → 搜尋 → 分析 → 綜整 → 撰寫」，審稿診斷橫切各階段皆可用，質性主題分析是處理一手資料的平行支線。從中段切入也可以（例如已有文獻池、直接從分析開始），各 skill 會檢查前置輸入、缺什麼就告訴你。
