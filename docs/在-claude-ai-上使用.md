# 在 claude.ai 上使用：把七支 skill 當獨立工具

這套工作流的完整形態是**一條管線**：七道工序共用一個持久的工作區資料夾，前一支的產出自動落在下一支讀得到的位置，這需要在 Claude Cowork 或 Claude Code環境下跑，也就是說需要付費版才能用。

但如果你只是想先小規模嘗試，這**七支 skill 在 claude.ai 網頁版都裝得起來，而且 Skills 從免費方案起就可用。** 差別在於：claude.ai 上你得到的不是一條管線，而是**七把獨立的工具**，今天開一個對話討論論文架構，明天開另一個對話分析一篇文獻，工序之間的交接由你手動搬檔完成，如果你已經有其他習慣的 AI 模型，把設計概念搬過去也能用。

這份文件就是寫給這種用法的，它不是「完整工作流的閹割版說明」，而是**單支獨立使用的操作手冊**。

但不管你用哪一支，我都認為 skill 1 是必備的，它產出的 paper-structure.md 是你研究的核心。當然如果你習慣自己寫，也可以不用 skill 1，我想表達的是，在研究工作流裡，有一套**清楚的整體研究架構與規劃**，才是一份好研究的核心與起點。

## 一、安裝（一次性）

1. **Settings → Capabilities**，開啟 **Code execution and file creation**。skill 需要程式碼執行環境才會出現。
2. **Customize → Skills** → 點「＋」→「Create skill」→「Upload a skill」。
3. 上傳 `skills/packaged/` 內的 `.skill` 檔。它本身就是 zip，**直接上傳，不需拆解**——`references/`（範本、推理框架）與 `scripts/` 會隨包完整上傳。

**最小起步與完整環境相同，不必七支全裝**，研究者最常用的應該會是：research-design-diagnosis（研究設計）＋ literature-analysis（逐篇分析），這兩支構成「先想清楚結構、再深度讀文獻」的完整閉環。單支用法下可再加一支 **review-diagnosis**（審稿診斷），它是七支之中最不依賴檔案治理的一支，上傳一份稿件就能用，這三支在 claude.ai 上的體驗與完整環境幾乎沒有差別。

介面選項可能隨版本調整，細節以官方說明為準。

## 二、一次性準備：把控制檔放進 Project

claude.ai 的 **Projects 在 Free 方案也可用**，而且 Project files 會進入 skill 的程式碼執行環境——**這是單支用法能成立的關鍵**。

建一個 Project（例如「我的研究」），把這幾份放進 Project files：

| 檔案 | 為什麼要放 | 誰會讀 |
|---|---|---|
| `about-me.md` | 你的目標、時程、寫作慣性自我申報 | skill 1、2、4、5 |
| `research-identity.md` | 理論版圖、方法論立場（決定 AI 對你說話的深度） | skill 1、3、5、7 |
| `writing-standards.md` | 禁止清單、引用格式、學術中文語氣 | skill 1、4、5、6、7 |
| `paper-structure.md` | **整條管線的中樞**（由 skill 1 產出後放進來） | skill 2、3、4、5、7 |

放進 Project 之後，這個 Project 內開的每一個對話都讀得到它們，**你不必每次重新上傳**。這一步做完，「單支獨立使用」就從麻煩變成順手。

沒有 Project 也能跑，只是每次對話都要把控制檔拖進附件。

## 三、skill 會自動偵測環境（v1.1.0 起）

skill 內文以 `00_專案控制/paper-structure.md` 這類**路徑**為錨點。claude.ai 上這些路徑不存在。

**自 v1.1.0 起，七支 skill 都會在啟動時先做環境偵測**：偵測到工作區資料夾不存在時，自動進入「無工作區模式」——改從對話附件與 Project files 尋找控制檔、產出改以下載檔交付、並在啟動宣告中告訴你「本次為無工作區模式」。你不需要做任何事。

**若 skill 仍回報找不到檔案**（例如你裝的是 v1.0.1 舊版，或偵測失效），在對話開頭補這段：

> 本次為 claude.ai 單支使用模式，**沒有工作區資料夾**。所有 `00_專案控制/` 的控制檔已放在 Project files（或本則訊息附件），請從那裡讀取，不要因為找不到資料夾路徑而中斷。所有產出請以**可下載檔案**交付，不要嘗試寫入 `05_輸出/`。

## 四、七支 skill 的單支使用卡

| # | Skill | 單支可用性 | 你要餵它什麼 | 它交給你什麼 |
|---|---|---|---|---|
| 1 | research-design-diagnosis | **完全可用** | Project files（控制檔）＋你的研究構想 | `paper-structure.md`（下載後放回 Project files） |
| 2 | literature-search | **自動搜尋不可用**（見第六節） | `paper-structure.md` | 僅手動匯出模式：你自己檢索、貼回 DOI，skill 續走分桶與匯入清單 |
| 3 | literature-analysis | **完全可用** | 一篇 PDF（對話附件） | 知識檔＋應用檔＋合併 .docx |
| 4 | literature-synthesis | **可用（有檔數上限）** | 同主題多篇的**知識檔＋應用檔**（skill 3 的產出） | 綜整知識檔＋綜整應用檔 |
| 5 | chapter-drafting | **可用（跨對話最痛）** | `paper-structure.md`＋該主題綜整檔＋`writing-standards.md` | 章節草稿 .md |
| 6 | review-diagnosis | **完全可用** | 你的稿件（PDF／docx） | 審查報告 .md／.docx |
| 7 | thematic-analysis | **可用（跨對話較痛）** | 去識別化逐字稿 | 編碼簿、主題地圖、主題敘事骨架 |

**「完全可用」的四支（1、3、6，加上規模不大時的 4）就是 claude.ai 的甜蜜點。** 它們天然是單次任務：一次一篇文獻、一次一份稿件、一次一輪結構討論。工作區資料夾能給的東西，它們本來就不太需要。

順帶釐清一個常見混淆：**「幫我綜整這篇文獻」其實是 skill 3（逐篇分析），不是 skill 4。** skill 4 吃的是「多篇已完成逐篇分析的產出」，做的是跨篇的理論版圖與研究缺口，單篇進 skill 4 沒有意義。

## 五、跨工序交接：你就是那條管線

在 Cowork 裡，skill 3 的產出落進 `05_輸出/`，你審定後搬進文獻庫，skill 4 自己去讀。在 claude.ai 裡，這條路徑由你手動走：

1. **skill 3 對話**：上傳 PDF → 拿到知識檔＋應用檔 → **下載**。
2. **你審定**：讀過、確認品質。這一步在兩個環境都一樣，是不可外包的簽核。
3. **skill 4 對話**：把審定過的多份知識檔／應用檔上傳（或放進 Project files）→ 拿到綜整檔 → 下載。
4. **skill 5 對話**：把綜整檔＋`paper-structure.md` 餵進去 → 拿到章節草稿。

麻煩的是搬檔，不麻煩的是決策，而**這套工作流真正的價值（AI 生產、你蓋章）完全保留了**。事實上，claude.ai 上的唯讀保護比 Cowork 更硬：AI 根本碰不到你電腦裡的任何檔案，誤寫是物理上不可能的，你付出的代價是自己當搬運工。

建議你在本機仍然照 `starter-kit/` 的 00–05 骨架整理下載檔，這樣日後若升級到 Cowork，整個工作區可以無痛接軌。

## 六、三個真實限制

### （一）literature-search 的自動搜尋不可用

claude.ai 沙盒的核准網域是**固定白名單**：PyPI、npm、GitHub、Ubuntu、crates.io 與 Anthropic API。它**不包含** `api.openalex.org` 與 `api.semanticscholar.org`，而 Free／Pro／Max 使用者無法自行增加網域。三支搜尋腳本必然被擋回 403。

skill 對此有內建處理：它會**宣告「環境阻擋」並轉入手動匯出模式**，而不是把 403 偽裝成「零筆結果」，更不會拿模型記憶捏造候選池。手動模式下你自己用 OpenAlex 網頁介面或 Zotero 檢索，把 DOI 貼回清單，續走分桶與匯入。

例外：**Team／Enterprise 的組織擁有者可自訂網域白名單**，把這兩個 API 加進去之後自動搜尋可以運作。

### （二）跨對話的狀態不續

skill 1、5、6、7 依賴 `05_輸出/progress-log.md` 做斷點續作；chapter-drafting 更以檔案為準保存「已凍結段落」，防止 AI 回頭改動你已經拍板的段落。claude.ai 沒有持久檔案，這些機制失效。

**替代動作**：每次續作的對話，把上一輪產出重新上傳，並在開場明講：

> 這是第 N 輪。前輪產出見附件，已凍結段落為第 X、Y 段，請從第 Z 段續作，不得改動凍結段。

受影響最重的是 **chapter-drafting（skill 5）** 與長資料集的 **thematic-analysis（skill 7）**，這兩支本來就設計成橫跨多個 session。實用建議：**一次對話跑完一節／一個階段**，不要跨對話拆得太碎。

### （三）檔案數上限

claude.ai 的對話附件有數量與容量上限（一般為每對話 20 個檔、單檔 30MB；Project files 的容量依方案而異）。這對 **skill 4（跨篇綜整）** 最敏感：15 篇文獻＝30 個檔，會超過單一對話的附件額度。

對策二選一：把同主題的知識檔**合併成一份 .md** 再上傳；或直接用 skill 4 內建的**分批綜整護欄**（每批 ≤10 篇先產中間摘要，最後合併）。skill 會在啟動時回報篇數與分批計畫。

## 七、什麼時候該升級到 Cowork／Claude Code

簡單來說，**這套工作流在 claude.ai 也能用**，只是比較卡，需要使用者作比較多動作。

但，如果你的用法是「**偶爾找 AI 討論研究架構、分析一篇文獻、審一份稿子**」，claude.ai 完全夠用，不必升級。這三件事正是這套工作流最不依賴檔案治理、也最有價值的部分。

出現以下任一情況，管線的價值才開始超過搬檔的成本：

- 你要跑 **literature-search** 的自動建池（唯一硬性理由）
- 你的知識檔累積到需要**跨專案複用**，檔案管理開始吃掉你的時間
- 你要用 **chapter-drafting** 實際寫完整章節（凍結段的跨 session 保護是這支的核心機制）
- 你已經確定要**綁定 Claude**，長期用這套工作流
